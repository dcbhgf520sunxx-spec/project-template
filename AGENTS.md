# 项目协作规则

本项目默认继承 `/Users/sunxinxin/Documents/Project/AGENTS.md` 的通用协作规则。

## 组件使用规则

本项目已经沉淀基础组件、业务组件和组件工作台。后续页面和功能实现时，默认优先调用已有组件，不在业务页面中临时自定义一套相同能力。

后续 AI 新增或改造业务功能时，必须先遵守 `docs/ai-development-rules.md`。该文档是业务页面调用样板和组件的执行入口，不能只按聊天中的临时判断实现。

涉及本地启动、数据库初始化、迁移命令和默认账号时，优先查看 `docs/数据库.md`。涉及数据库表结构说明时，同时维护 `docs/数据库表结构.md` 和 `docs/数据库表结构.xlsx`。

如果现有组件不能满足需求，先判断这是一次性页面差异，还是可复用能力：

- 一次性差异：在当前业务范围内做最小必要调整，不扩大成通用封装。
- 可复用能力：优先扩展或新增到组件层，并在业务代码中调用组件能力。
- 不确定是否应沉淀为组件时，先说明判断依据和影响范围，再向用户确认。

业务页面只负责组合组件、传入数据和处理业务流程；样式、交互、状态展示、空状态、加载态、错误态等通用能力，优先由组件承接，不在页面中重复实现。

新增或扩展组件能力后，必须同步考虑是否需要在组件工作台展示。通用按钮、切换、标签、抽屉、表格、表单等能力一旦沉淀到组件层，组件工作台应能看到入口或示例，避免只在业务页面里“隐形使用”。

## 页面样板接入规则

本项目已经沉淀页面样板时，后续业务模块默认以样板为基准接入，不重新设计一套页面结构。

实现新业务模块或改造旧业务模块时，先找到对应页面样板，沿用样板的布局结构、组件组合、筛选区、表格区、底部操作区、分页区、弹窗结构和交互节奏，只替换业务字段、接口数据和业务动作。具体接入要求以 `docs/ai-development-rules.md` 为准。

不得因为接入真实接口、赶进度或新建模块，就绕开既有样板重新拼页面。新增或改造业务模块时，不做“先实现再对照找差异”，应直接按既有页面样板和组件规则实现。

## 数据库表命名规则

数据库物理表名必须统一使用 `pms_` 前缀。新增表、初始化 SQL、后端查询 SQL、文档样板和测试用例都要按此前缀命名，例如 `pms_user`、`pms_role`、`pms_work_order`。

菜单 code、前端路由、接口路径和业务字段不是数据库物理表名，不因表名前缀规则改成 `pms_`。

新增或修改数据库结构时，必须同步处理以下位置：

- `backend/db/init/001_schema.sql`
- `backend/db/migrations/` 下的增量迁移
- 后端 SQL、接口和测试
- `docs/数据库表结构.md`
- `docs/数据库表结构.xlsx`

不能只改运行 SQL 或只改文档。涉及本地已有数据库时，还要说明需要执行哪条迁移。

数据库结构材料的分工固定如下：

- SQL / migration：给数据库执行，是技术和 AI 的真实结构来源。
- `docs/数据库表结构.md`：给技术和 AI 阅读、审查、diff。
- `docs/数据库表结构.xlsx`：给用户和业务同学核对字段，不要求用户阅读 SQL、migration 或 Markdown。

## 数据与接口规则

业务功能不能只写前端静态假数据。除组件工作台、纯视觉示例和明确的 mock 场景外，业务页面展示的数据、已读状态、操作结果等应接真实接口并落到数据库或已有持久化来源。

前端 API 层负责把后端字段转换为页面 record；业务组件负责展示和回调，不内置业务假数据。通用组件可以提供结构能力，但不应写死具体业务记录。

## 完成前验收规则

涉及业务页面改造时，完成前必须按页面类型做一次实际验收，不能只依赖构建通过。

列表页、详情页、新增编辑页的验收项以 `docs/ai-development-rules.md` 为准。涉及数据库或接口的功能，还要验证迁移、接口读写和页面真实渲染。

如果无法完成上述验收，必须明确说明未验收项和风险，不能笼统说“完成”。
