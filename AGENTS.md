# 项目协作规则

本项目默认继承 `/Users/sunxinxin/Documents/Project/AGENTS.md` 的通用协作规则。

## AI 交付链路

所有业务功能开发、优化、修复、重构、字段、接口、菜单、权限和样式变更，必须遵守 `docs/ai-delivery-flow.md`，不能只改局部代码后直接交付。

AI 负责实现、自动检查、迁移准备与发布执行；涉及数据库结构时，先由用户确认表结构设计，再由 AI 执行 migration。用户完成最终业务验收并确认发布后，AI 执行部署和线上健康检查。

## 组件使用规则

本项目已经沉淀基础组件、业务组件和组件工作台。后续页面和功能实现时，默认优先调用已有组件，不在业务页面中临时自定义一套相同能力。

后续 AI 进行任何业务页面变更时，必须先遵守 `docs/ai-development-rules.md`。该文档是业务页面调用样板和组件的执行入口，不能只按聊天中的临时判断实现。

AI 在判断底座没有某项能力、需要新增组件或扩展页面样板之前，必须先完成能力检索：核对 `docs/ai-development-rules.md` 中的用户说法与能力映射，继续查看对应组件入口、组件工作台实际示例和实际行为，不能只按用户口语或单一代码关键词搜索后下结论。判断结果必须说明已核对的组件入口与工作台示例；没有完成这条证据链时，不得宣称“底座不支持”。

开发新功能时，AI 在开始实现前必须先向用户输出简短的“组件支撑情况”，逐项说明功能需要、支持结论、拟使用的组件或页面样板、组件工作台示例和处理方式。没有输出并完成这项开工说明，不得直接开始实现业务页面；不涉及前端组件时也要明确说明。

支持结论为“部分支持”或“暂不支持”时，必须继续给出明确的建议结论，推荐“业务内最小处理”“扩展现有底座组件”或“新增底座组件”之一；不能只描述缺口或把选择留给后续实现。建议必须说明复用价值、组件职责和影响范围。低频、一次性且业务特有的能力，允许业务系统自行开发并选择“业务内最小处理”，不强制沉淀到底座；选择扩展或新增底座组件时，才需要先完善底座能力、组件工作台示例和测试，再由业务页面调用。

涉及本地启动、数据库初始化、迁移命令和默认账号时，优先查看 `docs/数据库.md`。涉及数据库表结构说明时，同时维护 `docs/数据库表结构.md` 和 `docs/数据库表结构.xlsx`。

如果现有组件不能满足需求，先判断这是一次性页面差异，还是可复用能力：

- 一次性差异：在当前业务范围内做最小必要调整，不扩大成通用封装。
- 可复用能力：优先扩展或新增到组件层，并在业务代码中调用组件能力。
- 不确定是否应沉淀为组件时，先说明判断依据和影响范围，再向用户确认。

业务页面只负责组合组件、传入数据和处理业务流程；样式、交互、状态展示、空状态、加载态、错误态等通用能力，优先由组件承接，不在页面中重复实现。

新增或扩展组件能力后，必须同步考虑是否需要在组件工作台展示。通用按钮、切换、标签、抽屉、表格、表单等能力一旦沉淀到组件层，组件工作台应能看到入口或示例，避免只在业务页面里“隐形使用”。

组件工作台各子页必须独立授权：概览、页面样板、基础规范、布局、基础组件、输入、反馈和数据展示均作为独立菜单权限；不得只凭父菜单一次性放开全部子页。权限键使用包含查询参数的完整地址，未授权用户不能通过修改 `category` 参数访问相邻分类。一级菜单固定保持“基础设置”位于“用户权限”之前。

## 页面样板接入规则

本项目已经沉淀页面样板时，后续业务模块默认以样板为基准接入，不重新设计一套页面结构。

实现新业务模块或优化既有业务模块时，先找到对应页面样板，沿用样板的布局结构、组件组合、筛选区、表格区、底部操作区、分页区、弹窗结构和交互节奏，只替换业务字段、接口数据和业务动作。具体接入要求以 `docs/ai-development-rules.md` 为准。

不得因为接入真实接口、赶进度或变更范围小，就绕开既有样板重新拼页面。所有业务页面变更不做“先实现再对照找差异”，应直接按既有页面样板和组件规则实现。

页面语义同样属于强制契约：列表操作必须进入 `OperationColumnActions` 并使用统一文字动作；删除使用 `DeleteConfirmAction`；状态变更使用 `StatusChangeAction` 或以 `StatusChangeAction` 为底层的业务封装，业务封装必须直接渲染公共组件，不能只靠组件命名绕过审计。有状态详情必须通过 `TemplateDetailPage.statusSection.items` 声明真实状态标签，由模板将同一组非空标签同步展示在标题后和右侧当前状态区；空值、空字符串和 `-` 只保留在右侧，不在标题后展示。业务页不得另传 `titleTags` 或通过 `statusSection.children` 自定义第二套状态样式。状态操作通过 `statusAction` 放在当前状态之后，不能用普通按钮冒充，也不能塞进右上角 `actions`。业务页面不得直接调用 `StatusFlowModal`，也不得恢复已删除的 `StatusFlowAction`。

## 数据库表命名规则

数据库物理表名必须统一使用 `pms_` 前缀。新增表、初始化 SQL、后端查询 SQL、文档样板和测试用例都要按此前缀命名，例如 `pms_user`、`pms_role`、`pms_work_order`。

菜单 code、前端路由、接口路径和业务字段不是数据库物理表名，不因表名前缀规则改成 `pms_`。

新增或修改数据库结构时，必须同步处理以下位置：

- `backend/db/init/001_schema.sql`
- `backend/db/migrations/` 下的增量迁移
- 后端 SQL、接口和测试
- `docs/数据库表结构.md`
- `docs/数据库表结构.xlsx`

不能只改运行 SQL 或只改文档。涉及本地已有数据库时，还要说明需要执行哪条迁移。

数据库结构材料的分工固定如下：

- SQL / migration：给数据库执行，是技术和 AI 的真实结构来源。
- `docs/数据库表结构.md`：给技术和 AI 阅读、审查、diff。
- `docs/数据库表结构.xlsx`：给用户和业务同学核对字段，不要求用户阅读 SQL、migration 或 Markdown。

## 数据与接口规则

业务功能不能只写前端静态假数据。除组件工作台、纯视觉示例和明确的 mock 场景外，业务页面展示的数据、已读状态、操作结果等应接真实接口并落到数据库或已有持久化来源。

前端 API 层负责把后端字段转换为页面 record；业务组件负责展示和回调，不内置业务假数据。通用组件可以提供结构能力，但不应写死具体业务记录。

## 完成前验收规则

涉及业务页面改造时，完成前必须按页面类型做一次实际验收，不能只依赖构建通过。

如果本次操作执行了 `npm ci`、依赖安装、构建或其他可能影响本地进程的命令，交付前必须重新检查前后端服务：确认 `3101` 健康检查返回 200、前端 `3102` 可访问；若服务停止，先按现有启动方式恢复，再交付可用地址。

列表页、详情页、新增编辑页的验收项以 `docs/ai-development-rules.md` 为准。涉及数据库或接口的功能，还要验证迁移、接口读写和页面真实渲染。

如果无法完成上述验收，必须明确说明未验收项和风险，不能笼统说“完成”。
