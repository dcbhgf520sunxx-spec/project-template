# 操作者身份修复同步说明

## 问题

旧实现由前端提交 `creator_id`、`updater_id`，后端直接写入数据库。登录用户可能被错误记录为管理员，也可以通过伪造请求冒充其他用户操作。

## 修复原则

- 前端写接口不再发送 `creator_id`、`updater_id`。
- 后端新增、编辑、删除、状态变更、批量操作和权限分配统一使用 `req.user.id`。
- 数据库的 `creator_id`、`updater_id` 以及操作日志 `user_id` 使用同一个登录用户ID。
- 即使旧客户端继续发送操作者字段，后端也忽略它们。

## 已覆盖模块

- 运维工单：创建、编辑、状态变更、批量指派、删除。
- 用户：创建、编辑、启停、重置密码、删除。
- 角色：创建、编辑、权限分配、删除。
- 基础档案：类型和记录的创建、编辑、启停、排序、删除。

## 已复制项目的同步方法

如果项目保留了与 `project-template` 相同的 Git 历史，优先从模板仓库获取修复提交后执行 `git cherry-pick`。如果业务项目已经大幅改造，则按以下顺序人工同步：

1. 删除前端 API 请求体中的 `creator_id`、`updater_id` 及写死的用户ID。
2. 在所有受 JWT 保护的后端写控制器中声明 `const operatorId = req.user.id`。
3. 新增时将 `creator_id`、`updater_id` 都写为 `operatorId`。
4. 更新、删除、状态变更、批量操作时将 `updater_id` 写为 `operatorId`。
5. 操作日志的 `user_id` 同样写为 `operatorId`。
6. 删除写接口校验规则中的 `creator_id`、`updater_id`。
7. 增加伪造操作者ID的回归测试，并运行完整构建和接口验证。

## 历史错误数据

代码修复只保证后续数据正确。不要执行类似 `UPDATE ... SET creator_id = 2 WHERE creator_id = 1` 的全量SQL，因为其中可能包含管理员真实创建的数据。

历史数据应依据访问日志、操作时间、业务单号和实际操作记录逐条确认。无法证明真实操作者的记录应保留原值，并单独标记为历史审计数据不确定，不能猜测修改。
