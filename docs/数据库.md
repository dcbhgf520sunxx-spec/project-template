# 本地启动与数据库初始化

## 文档分工

数据库相关材料分为四类：

- `backend/db/init/001_schema.sql`：全新数据库初始化脚本，给数据库执行。
- `backend/db/migrations/*.sql`：已有数据库升级脚本，给数据库执行。
- `docs/数据库表结构.md`：技术和 AI 阅读用，方便 Git diff 和代码审查。
- `docs/数据库表结构.xlsx`：用户和业务同学核对用，字段确认以这份表格最直观。

用户主要看 Excel 即可。AI 修改数据库结构时，必须同步维护 SQL、migration、Markdown 和 Excel，不能要求用户去理解 SQL 或迁移脚本。

## 环境

- Node.js：项目本地前后端使用 npm 脚本启动。
- 数据库：PostgreSQL 16，使用 Docker Compose。
- 数据库时区：统一使用 `Asia/Shanghai`。
- 默认后端端口：`3101`
- 默认前端端口：`3102`

## 启动 PostgreSQL

```bash
docker compose up -d postgres
```

数据库初始化脚本位于：

```text
backend/db/init/001_schema.sql
```

首次启动容器时会自动创建表结构、索引、默认菜单、默认角色、默认账号和少量样例数据。

```bash

npm install

```

```bash

```

```bash
cd frontend
npm install

```

访问：

```text
http://localhost:3102
```

## 重新初始化数据库

如果需要清空并重新执行初始化 SQL：

```bash
docker compose down -v
docker compose up -d postgres
```

注意：`down -v` 会删除本地 PostgreSQL 数据卷。

全新数据库由当前版本的 `001_schema.sql` 初始化后，应只登记现有 migration 基线，避免把已经合并进初始化脚本的历史变更重复执行：

```bash
cd backend
npm run db:migrate -- --baseline
```

`--baseline` 会先核对关键表、约束和索引确实存在，再登记历史 migration；它只允许用于由当前 `001_schema.sql` 新建的空白环境，禁止用于已有业务数据库。

## 已有数据库迁移

已有数据库不会自动重放 `backend/db/init/001_schema.sql`。所有增量 SQL 必须放到 `backend/db/migrations/`，由统一命令按文件名顺序执行，并记录到 `pms_migrations` 表。

先查看待执行项：

```bash
cd backend
npm run db:migrate -- --check
```

涉及新增表、字段、索引、关联或数据保留策略时，必须先由用户确认表结构设计；确认后再执行：

```bash
cd backend
npm run db:migrate -- --apply --user-approved
```

无参数运行 `npm run db:migrate` 时只会查看待执行项，不会修改数据库。真正执行迁移必须同时携带 `--apply` 和 `--user-approved`；缺少用户确认参数时命令会直接拒绝执行。

任一 migration 失败时，当前文件会回滚，后续文件不会执行。迁移完成后，AI 必须验证受影响接口的真实读写和页面渲染。
