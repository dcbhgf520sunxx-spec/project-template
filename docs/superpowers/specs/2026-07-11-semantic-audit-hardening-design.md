# 页面语义与接口契约审计补强设计

## 目标

补齐当前门禁只能检查“有没有使用组件”、不能充分检查“组件是否按正确语义使用”的缺口。新规则只强化底座结构契约，不把产品、项目、工单各自的业务流转写死到底座。

## 范围

本次只修改 `project-template`：

- 补强前端 JSX 语义审计。
- 增加业务状态组件溯源检查。
- 增加前端读取接口运行时契约检查。
- 提供业务状态流转测试样板和门禁要求。
- 更新 AI 开发规则与统一验证命令。

不修改 PMIS 产品、项目业务代码，不增加兼容组件，不生成一次性同步说明。

## 详情状态语义

有状态详情必须同时具备：

- `statusSection`：右侧当前状态信息。
- `titleTags`：标题后的状态标签，内容必须包含 `StatusTag` 或明确的状态标签业务封装。
- `statusAction`：当前状态后的操作入口，内容必须包含 `StatusConfirmAction`、`StatusChangeAction`，或以标准状态组件为底层的业务状态动作。

状态操作不得放进右上角 `actions`。仅写一个普通按钮、空节点或与状态无关的内容，不能满足门禁。

审计不根据普通字段名猜测页面是否有状态；当业务详情展示或操作状态时，代码必须显式声明 `statusSection`，以此形成稳定、可审计的页面契约。

## 业务状态组件溯源

允许业务模块创建 `ProjectStatusChangeAction`、`WorkOrderStatusChangeAction` 等封装，用于补充日期、原因、结果等业务字段。

放行条件：

- 业务组件名称以 `StatusChangeAction` 或 `StatusConfirmAction` 结尾。
- 其源码必须直接渲染公共 `StatusChangeAction` 或 `StatusConfirmAction`。
- 业务页面不得直接渲染底层 `StatusFlowModal`。
- 不采用按模块名称维护白名单的方式。

这样既允许业务扩展，也避免只改组件名字绕过审计。

## API 运行时契约

前端读取接口必须在 `unwrap()` 中传入运行时契约：

- 对象使用 `objectContract`。
- 数组使用 `arrayContract`，数组元素为对象时继续使用 `objectContract`。
- 分页结果必须校验分页字段和记录列表。

写入类接口允许以下明确返回不使用对象字段契约：

- `unwrap<null>()`。
- 不读取响应数据的删除、状态变更等操作。
- 仅返回 ID 的写入接口仍应使用对象契约校验 `id`。

审计范围为 `frontend/src/api`，组件工作台和测试文件不参与业务 API 审计。

## 业务流转规则

底座不硬编码“项目只能从未启动到进行中”之类业务规则。每个包含多状态流转的模块必须提供模块级测试，至少覆盖：

- 允许的来源状态和目标状态。
- 禁止的跨级或反向流转。
- 特定目标状态要求的附加字段，例如完成日期、暂停日期、处理结果。
- 前端可选目标与后端校验使用同一套业务定义或分别受到一致测试约束。

底座提供可复制的测试结构，并在 AI 开发规则和交付清单中要求执行。

## 测试与门禁

按测试驱动顺序实施：

1. 先增加能够复现现有漏洞的失败测试。
2. 最小修改审计脚本使测试通过。
3. 加入允许的正确写法测试，避免误伤标准页面。
4. 将新增测试加入 `scripts/verify-change.mjs`。
5. 使用兼容命令运行完整门禁：

```bash
PATH=/usr/local/bin:$PATH /usr/local/bin/node scripts/verify-change.mjs
```

6. 最后确认 `3101` 健康接口和 `3102` 前端均返回 200。

## 成功标准

- 普通按钮冒充 `statusAction` 会被阻断。
- 任意内容冒充 `titleTags` 会被阻断。
- 只靠 `*StatusChangeAction` 命名、内部未调用公共组件会被阻断。
- 读取接口缺少运行时契约会被阻断。
- 标准产品二态状态、项目多态状态和工单带附加字段状态均能通过正确实现扩展。
- 完整门禁、构建、后端测试和服务健康检查全部通过。
