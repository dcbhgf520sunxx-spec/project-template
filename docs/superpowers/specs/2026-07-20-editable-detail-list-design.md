# 可编辑明细表单设计

## 目标

在现有表单体系中补充一套可复用的“多行明细编辑”能力。业务页只声明字段、控件、校验和默认值，底座统一承接增删行、自动序号、最少行数和响应式布局。

## 适用场景

- 合同阶段、付款计划、预算明细、里程碑等需要一次编辑多条结构化数据的表单。
- 每一行包含多个可校验字段，并随主表单一次提交。
- 不适合纯展示列表；纯展示仍使用 `SearchTable` 或 `TemplateDetailTableSection`。

## 组件职责

公共组件 `AdminProFormEditableList` 负责：

- 电脑端使用表头和行式布局，形成表格化录入体验。
- 电脑端按实际列宽展示；列较多时中间字段横向滚动，并固定左侧序号和右侧操作。
- 窄屏自动切换为逐条卡片，不要求业务页维护第二套结构。
- 自动展示序号，序号不写入提交数据。
- 统一新增、删除和最少保留行数规则。
- 复用 `ProFormList`，让明细数组随 `TemplateFormPage` 一起校验和提交。
- 允许业务页通过列渲染函数继续使用现有 `AdminProForm*` 控件。

业务页面负责：

- 明细字段及中文表头。
- 每个字段使用的输入控件和校验规则。
- 新增行默认值。
- 业务接口、保存和持久化。

## 交互规则

- 默认至少保留一行；达到最少行数时删除不可用，并提示原因。
- 新增按钮固定在明细末尾，默认使用通用文案“新增”，仍允许按业务场景修改。
- 电脑端左侧固定序号、右侧固定操作；窄屏时取消固定，操作区放在卡片底部。
- 窄屏卡片中的每个字段显示对应中文标签。
- 最大行数由业务页按需声明，未声明时不额外限制。

## 接入边界

- 新增编辑页仍必须从 `TemplateFormPage` 接入。
- 业务模块不得直接使用 `ProFormList`、`Form.List` 或 `EditableProTable` 临时拼同类能力。
- 组件工作台必须提供可真实增删、校验的示例，作为后续 AI 能力检索证据。
- 本次不新增依赖，不修改数据库结构。
